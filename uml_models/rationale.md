## Data Storage System
The Data Storage System is centred around the DataStorage class, which acts as the main repository for all patient data. It stores Patient objects in a HashMap indexed by patient ID, making retrieval straightforward. DataStorage owns its patients through composition, and each Patient owns its PatientRecord objects the same way, meaning records cannot exist without a patient and patients cannot exist without the storage. PatientRecord is a simple immutable class holding a measurement value, record type, and timestamp. DataRetriever handles all read queries from medical staff, keeping retrieval logic separate from storage logic so that if the storage structure changes, the rest of the system is unaffected. AccessController works alongside DataRetriever to check whether a staff member has permission to access a patient's records before any query runs, which is important for maintaining patient privacy. DataDeletionPolicy manages how long data is kept by flagging and removing records older than a set number of days, preventing unnecessary data buildup. Overall the system is modular and straightforward to extend without major changes to existing components.

## Alert Generator System
The Alert Generation System is built around the AlertGenerator class, which serves as the core evaluator of incoming patient data. It holds a reference to DataStorage to retrieve patient records, and uses a list of AlertThreshold objects to define the conditions under which an alert should be triggered. Each AlertThreshold encapsulates a condition type, a threshold value, and an operator, meaning different patients can have different alert rules without any changes to the generator itself. When a breach is detected, AlertGenerator creates an Alert object containing the patient ID, condition, and timestamp, and passes it to AlertManager for dispatching. What is effective about this design is the clear separation of responsibilities: AlertGenerator only evaluates data and creates alerts, while AlertManager handles all dispatching logic and notifies the relevant MedicalStaff. This means if the notification system changes — for example adding email or SMS — only AlertManager needs to be updated. Similarly, if new threshold types are needed, only AlertThreshold is modified. The aggregation between AlertManager and Alert reflects that alerts exist independently and can be resolved or archived without being destroyed. Overall the system is modular, easy to extend, and reflects realistic hospital alert logic where different patients have personalized thresholds and alerts are routed to the appropriate medical staff.
